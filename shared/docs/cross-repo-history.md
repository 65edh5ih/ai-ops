# クロスリポジトリ履歴の書き方

Notion「AI Cross-Repo Task Log」に**共通基盤（`shared/` 由来ファイル・共通ブロックの配線・consumer-template 由来 workflow 等、
複数リポジトリに同型で存在するもの）の対応履歴**を残すときの書き方。
*いつ*残すか（共通基盤を触ったら固有履歴に加えてここにも残す）は共通ブロック（`AGENTS.md` の「作業完了時の必須手順」）側にある。
このドキュメントは*どう書くか*だけを扱う。実際にクロスリポジトリ履歴を書くときに読む。

## なぜ書き方に注意が要るか

この履歴は**別の consumer から読まれる**。読む側の手元からは、書いた側のリポジトリ自体が見えない
（PR 番号・URL・`owner/repo#n` も解決できない）。**各 consumer に届いているのは同期された成果物だけ**なので、
手がかりは**読む側の手元にあるもの**に置く必要がある。

## 書き方ルール

- 各エントリに「**確認先（手元で grep）**」を付け、読む側が自分のリポジトリのファイル・文字列で現行/旧状態を
  判別できるようにする（例: 自分の `AGENTS.md` 共通ブロックの該当節、`.github/workflows/*.yml` 内の特定行）。
- **PR 番号・URL・commit SHA は主たる手がかりにしない**。書くなら末尾に「（書いた側にアクセスできる人向け・
  consumer からは解決不可）」と明記して隔離する。
- 内容は「**なぜ・どう直したか**」（コードに無い判断根拠）を中心に。「何を変えたか」は同期成果物を見れば分かる。

## このルール自体が改廃対象

実際に履歴を読み書きして「この手がかりでは辿れなかった／別の形が要る」と分かったら、**促されるのを待たず**この
ドキュメントを直す（共通ブロックの必須手順「既存ドキュメントの実証的訂正」の具体適用）。包括ルールに委ねず明示するのは、
抽象的な改廃義務だけでは自主的に発火しなかった実例があるため。

> このドキュメントは ai-ops が `shared/docs/cross-repo-history.md` を正本として配布している（consumer では
> `docs/cross-repo-history.md`）。consumer 側で手編集しても次回同期で上書きされる。直すには outbox の
> `種別: shared-file` / `対象パス: docs/cross-repo-history.md` 提案を出すか、横断的・構造的な修正なら
> Notion「AI Cross-Repo Task Log」に依頼を出して ai-ops 側で一括編集する（→ `docs/outbox-proposal.md`）。
