# タスク履歴の規約（全リポジトリ共通）

タスク履歴の**目的は2つ**: (1) スレッド（セッション）間で「さっきのあれ」を通じさせる**短期記憶**、
(2) コードに残らない判断根拠（「なぜ」）の**恒久保管**。(1) のために直近分は小さく保ち、
(2) のために古い分はアーカイブへ移して捨てない。

## 場所（全リポジトリ共通）

- 履歴本体: **`docs/AI_TASK_HISTORY.md`**（新しいエントリが上）
- アーカイブ: **`docs/history-archive/<YYYY>.md`**（年ごと。こちらも新しいエントリが上）

## 書き方

- **1タスク（スレッド）= 1エントリ**。完了手順の一部として、ユーザーへの完了報告より前に書く。
- 見出しは `## YYYY-MM-DD 短いタイトル`（同じ日に複数エントリがあってよい）。
- 中身は「**なぜ**」（コードに無い制約・判断根拠・ハマりどころ）を数行。「何を変えたか」は
  コード・PR で分かるので書かない。
- 「さっきのあれ」の照合に使われるので、タイトルにはユーザーとのやりとりで使われた呼び名・
  対象名をそのまま入れる。
- **他の恒久 doc から履歴の特定エントリへ「日付付き」でリンクしない**（例:
  「`docs/AI_TASK_HISTORY.md` の YYYY-MM-DD のエントリ参照」）。エントリは下記アーカイブで
  `docs/history-archive/<YYYY>.md` へ移動するため、日付付きポインタは移動後に壊れる。経緯を
  恒久 doc に残したいなら、その事実を doc 内に書き切る（履歴へのリンクに頼らない）。履歴ファイルの
  「場所そのもの」を説明する言及（本規約・`docs/README.md` 等）は対象外。

## アーカイブ（自動 — エージェントは何もしない）

- 保持量: 履歴本体に残すのは**エントリのある日付で数えて直近2日分**（同日の複数エントリはまとめて
  1日と数える。暦日ではなく作業日: 間に作業のない日を挟んでも、エントリのある直近2日分を残す）。
  件数でなく日数で数えるのは、短期記憶の必要量が経過時間で決まるため。
- 3日分以上になった古い日付のエントリは、**ai-ops の集中バッチ**（archive-task-history workflow・
  1日1回）が `docs/history-archive/<YYYY>.md` の先頭へ丸ごと移す PR を自動生成・マージする
  （体裁はそのまま・書き直さない）。
- エージェントは**追記だけ**すればよい（MUST NOT: セッション内で手動アーカイブしない。
  超過に気づいても放置してよい——決定的なファイル移動にトークンを使わない。次のバッチが処理する）。
- アーカイブはオンデマンド層: 過去の経緯が要るときだけ読む・grep する。

## 各リポジトリ側に書くこと

- 各 AGENTS.md の固有パートに書くのは**この規約からの逸脱（例外）だけ**。原則、何も書かなくてよい。
- この規約自体を直したいときは outbox の `shared-file` 提案（`対象パス: docs/task-history.md`）。
  → `docs/outbox-proposal.md`
