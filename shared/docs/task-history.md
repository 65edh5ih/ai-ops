# タスク履歴の規約（全リポジトリ共通）

タスク履歴の**目的は2つ**: (1) スレッド（セッション）間で「さっきのあれ」を通じさせる**短期記憶**、
(2) コードに残らない判断根拠（「なぜ」）の**恒久保管**。(1) のために直近分は小さく保ち、
(2) のために古い分はアーカイブへ移して捨てない。

## 場所（全リポジトリ共通）

エントリは**書く場所と読む場所が分かれている**（並行 PR のコンフリクトを避けるため。理由は下記「なぜ
フラグメント方式か」）:

- **書く**（エージェント）: **`docs/history-inbox/<YYYY-MM-DD>-<スラッグ>.md`** に**新規ファイル1つ**として
  置く（＝未統合フラグメント。既存ファイルには一切追記しない）。
- **統合先**: **`docs/AI_TASK_HISTORY.md`**（新しいエントリが上）。inbox のフラグメントを ai-ops の集中
  バッチがここへ取り込む（下記「統合とアーカイブ」）。エージェントはこのファイルを直接編集しない。
- **アーカイブ**: **`docs/history-archive/<YYYY>.md`**（年ごと。こちらも新しいエントリが上）。

「さっきのあれ」を**読む**ときは、`docs/AI_TASK_HISTORY.md`（統合済み・直近2作業日分）**と
`docs/history-inbox/` 内の全ファイル**（未統合の最新分）の**両方**を見る。より古い経緯は
`docs/history-archive/` を grep。

## 書き方（フラグメントを1ファイル置く）

- **1タスク（スレッド）= 1フラグメント = 1エントリ**。完了手順の一部として、ユーザーへの完了報告より
  前に置く。
- 置き場所は `docs/history-inbox/<YYYY-MM-DD>-<スラッグ>.md`。ファイルの中身は**エントリ本体だけ**
  （`## YYYY-MM-DD 短いタイトル` の見出し＋数行の本文。`# ` の H1 やその他の前置きを書かない）。
- **ファイル名は他の並行セッションと必ず重複させない**（MUST）。同名ファイルを別々のブランチで作ると
  マージ時に add/add コンフリクトになり、フラグメント方式の意味が失われる。`<スラッグ>` は**作業中の
  ブランチ名**から採る（このリポジトリ運用のブランチ名は末尾にランダム片を持つため一意になる）。
  例: ブランチ `claude/foo-bar-xy5waa` → `docs/history-inbox/2026-07-20-foo-bar-xy5waa.md`。
- 見出しは `## YYYY-MM-DD 短いタイトル`。中身は「**なぜ**」（コードに無い制約・判断根拠・ハマりどころ）を
  数行。「何を変えたか」はコード・PR で分かるので書かない。
- 「さっきのあれ」の照合に使われるので、タイトルにはユーザーとのやりとりで使われた呼び名・
  対象名をそのまま入れる。
- **他の恒久 doc から履歴の特定エントリへ「日付付き」でリンクしない**（例:
  「`docs/AI_TASK_HISTORY.md` の YYYY-MM-DD のエントリ参照」）。エントリは下記アーカイブで
  `docs/history-archive/<YYYY>.md` へ移動するため、日付付きポインタは移動後に壊れる。経緯を
  恒久 doc に残したいなら、その事実を doc 内に書き切る（履歴へのリンクに頼らない）。履歴ファイルの
  「場所そのもの」を説明する言及（本規約・`docs/README.md` 等）は対象外。

### なぜフラグメント方式か

`docs/AI_TASK_HISTORY.md` は「新しいエントリが上」なので、全セッションが**同じ行位置（先頭）に挿入**
する。並行 PR が2つあればほぼ確実にコンフリクトし、アーカイブバッチの自動 PR とも衝突しうる。
1エントリ＝1新規ファイルにすれば、各セッションが**別々のパスに触れる**だけになり衝突が原理的に消える
（changelog の towncrier / changelog.d と同型）。本体ファイルの書き手を集中バッチ1つに絞れる副次効果もある。

## 統合とアーカイブ（自動 — エージェントは何もしない）

- **統合（consolidate）**: ai-ops の集中バッチ（archive-task-history workflow）が `docs/history-inbox/` の
  全フラグメントを `docs/AI_TASK_HISTORY.md` の先頭へ日付順で取り込み、取り込んだフラグメントファイルを
  削除する（本文はそのまま・書き直さない）。統合されるまでの間は上記「読む」のとおり両方を見れば最新も拾える。
- **保持量**: 統合後、本体に残すのは**エントリのある日付で数えて直近2日分**（同日の複数エントリはまとめて
  1日と数える。暦日ではなく作業日: 間に作業のない日を挟んでも、エントリのある直近2日分を残す）。
  件数でなく日数で数えるのは、短期記憶の必要量が経過時間で決まるため。
- **アーカイブ（archive）**: 3日分以上になった古い日付のエントリを、同じバッチが
  `docs/history-archive/<YYYY>.md` の先頭へ丸ごと移す。統合・アーカイブとも1つの PR を自動生成・マージする。
- エージェントは**フラグメントを1つ置くだけ**でよい（MUST NOT: セッション内で本体への統合・手動
  アーカイブをしない。超過に気づいても放置してよい——決定的なファイル移動にトークンを使わない。
  次のバッチが処理する）。
- アーカイブはオンデマンド層: 過去の経緯が要るときだけ読む・grep する。

## 各リポジトリ側に書くこと

- 各 AGENTS.md の固有パートに書くのは**この規約からの逸脱（例外）だけ**。原則、何も書かなくてよい。
- この規約自体を直したいときは outbox の `shared-file` 提案（`対象パス: docs/task-history.md`）。
  → `docs/outbox-proposal.md`
