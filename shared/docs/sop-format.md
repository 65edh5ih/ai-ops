# 手順系 doc（SOP）の書式規約（全リポジトリ共通）

エージェント（Claude Code / Codex）に実行させる**手順書（SOP: Standard Operating Procedure）**の書き方。
書き手ごとの流儀で書かれた手順 doc は、実行するエージェント側で「どこまでが必須か」「何をもって完了か」の
解釈ぶれを生む。この規約で**制約の強さ**と**完了条件**を機械的に読める形に揃える。

## 適用範囲

- **共通・リポジトリ固有を問わず**、「エージェントに実行させる手順」を書いた doc すべて
  （`docs/` 配下が主。例: リリース手順・障害調査手順・データ移行手順・この規約群のような運用手順）。
- 適用外: 設計 doc（判断根拠の記録）・タスク履歴・README・コード内コメント。手順を含まない文書に
  無理に適用しない。
- 既存 doc は**触れたついでに準拠させる**（一斉変換はしない。改訂で開いたときが変換のタイミング。
  ただし別途タスクとして依頼された場合はまとめて変換してよい）。

## 書式（この順で）

1. **タイトルと目的**: 何を達成する手順か、1〜2行。
2. **いつ使うか（トリガ）**: どんな状況でこの doc を開くべきか。エージェントが「今がそのときだ」と
   判定できる具体性で書く（「必要に応じて」は書かない）。
3. **前提・パラメータ**: 実行に必要な入力・条件を列挙し、本文では `<owner/repo>`・`<ブランチ名>` のような
   プレースホルダで参照する（実値をハードコードしない。リポジトリ固有 doc で値が固定なら既定値として添える）。
4. **手順**: 番号付きステップ。各ステップに**検証可能な完了条件**（コマンドの期待出力・作られるファイル・
   PR の状態など、確認できる事実）を含める。
5. **よくある失敗**（任意）: 実際に起きたつまずきと対処。起きてから足す（想像で網羅しない）。

## 制約キーワード（RFC 2119）

手順文中の要求の強さは大文字キーワードで明示する（日本語文中でもそのまま使う）:

- **MUST / MUST NOT**: 逸脱したら壊れる・やり直しになる制約。
- **SHOULD / SHOULD NOT**: 原則従う。逸脱するときは理由をタスク履歴に記録する。
- **MAY**: 任意。
- キーワードの付いていない文は説明であって要求ではない、という前提で読まれる（そのつもりで書く）。

## ツール中立（複数エージェントが実行する前提で書く）

配布される手順は、複数の異なるエージェント（Claude Code / Codex / Gemini CLI / OpenHands 等）が実行する。
書き手が自分の使うツールを前提にすると、他エージェントで抜けが出る。手順を**特定ツールの有無・振る舞いに
紐づけない**:

- **要求するのは「能力」であって特定ツールではない**（MUST）: ステップは*何を*満たすか（例: 「対象リポジトリの
  workflow を dispatch できる」「`ci-logs` を読める」）で書く。*何で*満たすか（GitHub の MCP ツール・`gh` CLI・
  REST API 等）はランタイム依存の補足に留め、手順の本筋にしない。
- **判断・分岐を「ツールの有無」に紐づけない**（MUST NOT）: 「ツール X を持たないエージェントは代わりに Y する」
  という分岐は、本来の判断軸（可視性・機微性・安全性など）を*ツール能力*にすり替える。ある能力を満たせない
  ときは、別の振る舞いへ勝手に切り替えず**停止してユーザーに依頼する**（MUST）。
  - 実例: net-fetch のモード選択で「`add_repo` を持たないエージェントは分散モードを既定にする」と書いたため、
    可視性・機微性で選ぶ原則が崩れた（能力でモードが決まってしまった）。正しくは「集約が正しいのに実行能力が
    無いなら停止して依頼」。
- **同じ要求は全ツール経路に効くよう書く**（MUST）: RFC 2119 キーワードを一部の経路にだけ付けない。無キーワード
  文は「説明であって要求ではない」と読まれる（上記「制約キーワード」）ため、1経路にだけ MUST を付けると他経路
  では要求が消える。**要求そのものは経路によらない MUST として書き、経路ごとの満たし方はその MUST の下の補足に
  置く**。
  - 実例: net-fetch の dispatch で「既定ブランチの ref で回す」が REST 経路にだけ MUST で、`gh`/MCP 経路は
    無キーワードだった。→ 別ブランチ ref でレビュー前の workflow が走りうる穴になった。正しくは全経路共通の
    MUST に格上げし、満たし方（REST=body の `ref`／`gh`=`--ref`／MCP=ref パラメータ）を補足に分ける。

## 淘汰（書式と同じ重みの規約）

- 実行して実態と違うと分かった箇所は**その場で直す**（共通 doc は outbox の `shared-file` 提案、
  リポジトリ固有 doc は直接編集）。黙って回避策で済ませない。
- 使われなくなった SOP は削除を提案する。「いつか使うかも」で残さない
  （手順は腐り、腐った手順は従われないより悪い——半分だけ従われる）。

## Skills ラッパー（発火の自動化）

共通 SOP は doc 本体（`docs/<name>.md` ＝正本）に加えて、各エージェントが該当タスクで自動発火
できる skill ラッパーが同期配布される（SKILL.md は共通標準形式。中身はどのパスも同一）:

- `.claude/skills/<name>/SKILL.md`（Claude Code）
- `.codex/skills/<name>/SKILL.md`（Codex）
- `.openhands/skills/<name>/SKILL.md`（OpenHands V1）
- `.gemini/skills/<name>/SKILL.md`（Gemini CLI。AGENTS.md 本体は `GEMINI.md -> AGENTS.md` 入口 symlink で読む）
- `.agents/skills/<name>/SKILL.md`（Antigravity）

ラッパーは「`docs/<name>.md` を読んで従え」の薄いポインタで、frontmatter の `description` が発火トリガ。
**SKILL.md に手順本体を書かない**（正本は常に `docs/` 側。二重管理にしない）。配布物なので手編集しない
（直したいときは outbox 提案）。

`.claude/skills/` 以外の4パスは、ai-ops の配布スクリプトが正本 `.claude/skills/` から**配布時に
自動生成**する。共通 SOP に skill ラッパーを添えるときに書く（提案する）のは
`shared/.claude/skills/<name>/SKILL.md` の1ファイルだけ（ミラーを手で作らない。MUST NOT）。

リポジトリ固有の SOP にも、同じパターンでローカルに skill ラッパーを置いてよい（MAY。本体を `docs/` に置き、
各 `.＜agent＞/skills/` にポインタだけ、という分担は共通 SOP と同じにする）。
