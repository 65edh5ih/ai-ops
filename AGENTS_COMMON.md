**作業前に必ず [`docs/DESIGN.md`](docs/DESIGN.md) を読むこと。** 共通基盤（ai-ops）の全体像（下り＝配布／上り＝提案、各ファイルの唯一の書き手、トークン構成）が書いてある。

## 作業完了時の必須手順（順番厳守・ユーザー返答より前に）

1. **タスク履歴を更新する**（このリポジトリの履歴ファイル）。残すのは「**なぜ**その変更をしたか」
   — コードを読めば分かる「何を変えたか」ではなく、コードに無い制約・判断根拠。
2. **再利用可能なノウハウを該当ドキュメントへ反映する**。特に**既存ドキュメントの手順が実証で
   誤り・不足だと分かったら、その場で正す**（黙って回避策で済ませない）。
   **ユーザーに促される前に、自分から完了手順の一部として行う。**
3. **共通基盤に関わる変更なら、その対応履歴を共通基盤側にも残す**。`shared/` 配下のファイル・
   共通ブロックの配線・consumer-template 由来の workflow など、**複数リポジトリに同型で存在する
   「共通基盤」**を触ったときは、リポジトリ固有の履歴ファイルだけに書くと、同じ基盤を別 consumer で
   直すとき（例: nikki-san で同じ修正）に経緯が見えない。固有履歴に加えて、**クロスリポジトリの
   共通履歴置き場（下記「クロスリポジトリ情報引き継ぎ（Notion）」の DB、ユーザー指示時に読み書き）にも
   「なぜ・どう直したか」を残す**。※`.ai-ops/outbox/` は `AGENTS_COMMON.md` を丸ごと置換する経路なので
   **履歴の記録には使わない**（共通ブロックを壊す）。記録先の有無・運用はリポジトリ／ai-ops 側の固有設定に従う。

> 履歴ファイルの場所・保持件数など、リポジトリ固有の設定は各 AGENTS.md の固有パートで定義する。

## 新情報・ノウハウの記録先の判断

- **常時ロード層（AGENTS.md）に書く**: どのタスクでも必要なもの（デプロイ先・環境変数・外部連携の概要・
  ローカルで再現できない制約）。全タスクのコストになるので最小限に。
- **オンデマンド層（docs/ 等）に書く**: 特定タスクでのみ必要なもの（手順・設計判断・障害調査）。
- 判断基準: 「次のタスクが何であれ必要か？」NO なら常時ロード層に置かない。
- 新規追記だけでなく「**既存ドキュメントの実証的訂正**」も同格の対象。

## ブランチ・PR の規律

- **1依頼 = 1PR**。依頼ごとに最新の `main` から新ブランチを切る。別依頼のコミットを既存ブランチに積まない。
- **ブランチ作成前に必ず `git fetch origin main`**（コンフリクト予防）。
- **既存ブランチへ push する前に、その PR がマージ済みでないか必ず確認する**（無条件）。
  正本は PR の実マージ状態（`merged`/`state`）。`git log origin/main..HEAD` は squash merge を検出
  できないので正本にしない。マージ済み/クローズ済みなら push せず新ブランチを切り直す。
- URL に影響する変更・外向き操作は必ずレビュー/確認を求める。

## クロスリポジトリ情報引き継ぎ（Notion）

- **タスク固有の引き継ぎ**（「リポジトリ X と同じことを Y で」）にのみ Notion を使う。
  タスクに必要と判断したときは自律的に参照してよい。記録（書き込み）はユーザー指示時、または共通基盤の対応履歴を残す場合に行う。
- **共通基盤（`shared/`・consumer-template 由来 workflow 等）の対応履歴**のうち、リポジトリをまたいで
  見えるべきものは Notion のクロスリポジトリ履歴 DB（"AI Cross-Repo Task Log"）に残す。履歴を運ぶ同期経路は
  他に無く、`.ai-ops/outbox/` は共通ブロックを置換するため履歴には使えないため、ここが正規の置き場。
- 恒常的な共通運用ルールは Notion ではなく **ai-ops（このブロック）** が正。手動リレーしない。
- **クロスリポジトリ履歴の書き方（重要）**: この履歴は**別の consumer から読まれる**。読む側の手元からは
  書いた側のリポジトリ自体が見えない（PR 番号・URL・`owner/repo#n` も解決できない）。**各 consumer に届いて
  いるのは同期された成果物だけ**なので、手がかりは**読む側の手元にあるもの**に置く:
  - 各エントリに「**確認先（手元で grep）**」を付け、読む側が自分のリポジトリのファイル・文字列で現行/旧状態を
    判別できるようにする（例: 自分の `AGENTS.md` 共通ブロックの該当節、`.github/workflows/*.yml` 内の特定行）。
  - **PR 番号・URL・commit SHA は主たる手がかりにしない**。書くなら末尾に「（書いた側にアクセスできる人向け・
    consumer からは解決不可）」と明記して隔離する。
  - 内容は「**なぜ・どう直したか**」（コードに無い判断根拠）を中心に。「何を変えたか」は同期成果物を見れば分かる。
  - **この書き方ルール自体が改廃対象**: 実際に履歴を読み書きして「この手がかりでは辿れなかった／別の形が要る」と
    分かったら、**促されるのを待たず**ここを直す（冒頭の必須手順2の具体適用。包括ルールに委ねず、この節で明示する
    のは、抽象的な改廃義務だけでは自主的に発火しなかった実例があるため）。

## この共通ブロック自体を直したくなったとき（重要）

このマーカー区間は各リポジトリで手編集しても**次の同期で上書きされる**。共通ルールの追加・修正に気づいたら、
**作業中のリポジトリの `.ai-ops/outbox/` に提案ファイルを置く**（マーカー区間は触らない）。これが consumer→ai-ops の唯一の上り口で、
転記もドリフトも起きない（ai-ops が単一の正のまま）。

- ファイル名: `YYYY-MM-DDThhmmss-<短い説明>.md`（先頭の時刻で処理順が決まる）。
- **必ず先頭に frontmatter を付けて種別を明示する**（種別不明の提案は collect が安全のためスキップする）。
- このファイルを consumer の `main` に載せれば（通常の PR でよい）、ai-ops の collect workflow が拾って
  取り込み PR を自動生成する。オーナーがマージすると全 consumer へ配布される。
- 各リポジトリの `AGENTS.md` のマーカー区間を直接書き換えない（上書きされて消えるだけでなく、ドリフトの原因になる）。

**共通ルール（テキスト）の提案** — frontmatter + 共通ブロック全文:

```markdown
---
種別: common-block-edit
---
（ここにマーカー区間の中身を全文コピーし、必要な変更を加える。差分ではなく全文を置く）
```

**共有実ファイル（shared/）の提案** — frontmatter + ファイル本文:

```markdown
---
種別: shared-file
対象パス: .github/actions/my-action/action.yml
---
（ここにファイルの全文を書く）
```

## 共通か固有かの判断（「これは他リポジトリでも要る」と気づいたとき）

作業中に再利用可能な知見・ルール・インフラに気づいたら、まず **「全リポジトリで同一内容が要るか？」** を問い、出し先を振り分ける:

- **全リポジトリ共通の振る舞いルール（テキスト）** → ai-ops が正本。`.ai-ops/outbox/` に提案を置く（上の手順）。
  このリポジトリの `AGENTS.md` を直接書かない。
- **このリポジトリ固有**（他では違う／片方だけ必要）→ このリポジトリのドキュメントや `AGENTS.md` の固有パートに書く。
- **全リポジトリでバイト一致すべき実ファイル**（composite action・共有スクリプト等）→ ai-ops の `shared/` が正本。
  上の手順で `種別: shared-file` / `対象パス: <shared/ 配下の相対パス>` frontmatter を付けた提案ファイルを outbox に置く。
  collect が `shared/<対象パス>` へ配置 PR を生成し、マージ後に既存の sync で全 consumer へ再配布される。
  このリポジトリだけに勝手に置かない（ドリフトの元）。
- **共通だが一部が固有** → 共通部分だけ上り提案し、固有部分はこのリポジトリに薄く残す
  （例: 「新規 workflow に publish-ci-logs を組み込む」義務は共通、collector 名やドキュメントの場所は固有）。

## CI ログ運用（新規ワークフローを追加するとき）

CI ログは AI エージェント専用なので `main` を汚さない専用ブランチ `ci-logs`（main から分岐・orphan）へ slice 単位で publish する。
このための composite action `.github/actions/publish-ci-logs` は **ai-ops が全 consumer へ配布する共通インフラ**（`shared/` 同期。手で編集しない）。
**新規に `.github/workflows/` を追加するときは必ず CI ログ出力を組み込む**:

1. ジョブに `permissions: contents: write` を付ける。
2. スクリプトログを `logs/ci/scripts/<name>.log` へ出す（`2>&1 | tee`）。
3. ジョブ末尾に「Stage CI log snapshot」と「Publish logs to ci-logs branch」(`uses: ./.github/actions/publish-ci-logs`) の2ステップを `if: always()` で足す。
4. リポジトリにログ collector（`workflow_run` で集約する workflow）があるなら、その `workflows` リストにワークフロー名を登録する。
5. リポジトリにログ設計ドキュメントがあるなら、その slice 一覧テーブルに行を足す。

> collector・設計ドキュメントの**ファイル名や有無はリポジトリ固有**（各 `AGENTS.md` の固有パートに書く）。
> 上記の「publish-ci-logs を組み込む」という義務自体は全リポジトリ共通。
