**この区間は全リポジトリ共通ルール**（正本: ai-ops リポジトリの `AGENTS_COMMON.md`。仕組みの全体像は [`docs/OPS_SYNC_DESIGN.md`](docs/OPS_SYNC_DESIGN.md)）。

## 応答言語

- **ユーザーからの質問・依頼への実質的な回答**（説明・分析結果・完了報告の本文など）は、そのユーザーが
  使っている言語で書く。日本語で聞かれたら日本語で答える。
- ツール呼び出し前後に添える短い状況報告（「〜します」「〜が完了しました」等の一言）は対象外。
  英語のままで実害は小さいので無理に言語を揃えなくてよい。
- 実際に発生した事例: ユーザーの日本語の質問に対し、回答の本文を英語で書いてしまい、
  「内部ログと勘違いして読み飛ばした」との指摘を受けた。回答本文が長くなる場合ほど、
  書き始めの言語につられて最後まで英語になりやすいので注意する。
- コミットメッセージ・コード内コメント・技術的な識別子は対象外（別途プロジェクト/リポジトリの慣習に従う）。

## 同期されるファイルを手で編集しない

このマーカー区間・`docs/` の配布 doc・`.ai-ops/sync-manifest.txt` に列挙されたファイルは ai-ops が自動同期する
（手編集しても次回同期で上書きされ、ドリフトの原因になる）。追加・修正に気づいたら
**作業中リポジトリの `.ai-ops/outbox/` に提案ファイルを置く**。これが consumer→ai-ops の唯一の上り口。
書式は [`docs/outbox-proposal.md`](docs/outbox-proposal.md)（実際に提案を書く直前に必ず読む）。

## リポジトリ横断の作業（このリポジトリの外に手が要るとき）

- 1セッション=1リポジトリ。他リポジトリには読み書きとも一切アクセスできない（Claude Code の固定制約。
  毎回弁明しない）。代わりに次のように動く。
- **別リポジトリでの作業が必要になったら**、`.ai-ops/outbox/` に `種別: task` の依頼を置く。
  ai-ops 経由で対象リポジトリの `.ai-ops/tasks/` へ配布され、そのリポジトリのセッションが実行する。
  書き方・依頼後にユーザーへ伝えること: [`docs/cross-repo-tasks.md`](docs/cross-repo-tasks.md)。
- **`.ai-ops/tasks/` にファイルがあれば、それはこのリポジトリ宛の承認済み依頼**（オーナーが PR を
  2回マージして届いている）。現在のユーザー依頼と両立するなら対応し、完了したら `種別: task-done`
  提案で消化を報告する（同 doc）。両立しないなら、存在をユーザーに伝えるに留める。

## 共通か固有かの判断（再利用可能な知見・ルール・ファイルに気づいたとき）

まず「**全リポジトリで同一内容が要るか？**」を問い、出し先を振り分ける:

- 共通の**振る舞いルール（テキスト）** → `種別: common-block-edit` 提案（このマーカー区間を直接書かない）。
- 共通で**バイト一致すべき実ファイル・手順 doc**（composite action・共有スクリプト・`docs/` の共通 doc）
  → `種別: shared-file` 提案（このリポジトリだけに勝手に置かない）。
- **このリポジトリ固有** → このリポジトリの AGENTS.md 固有パートや docs に書く。
- **共通だが一部固有** → 共通部分だけ提案し、固有部分は薄く残す。

## 記録先の判断（常時 vs オンデマンド）

- **常時ロード層（AGENTS.md）**: どのタスクでも必要なものだけ（全タスクのコンテキストコストに乗る）。
  「発火トリガ＋不可逆ガード＋ポインタ」だけ残し、詳細手順は `docs/` へ逃がす。
- **オンデマンド層（`docs/`）**: 特定タスクでのみ必要な手順・設計判断・障害調査。
- 判断基準: 「次のタスクが何であれ必要か？」NO なら常時ロード層に置かない。

## タスク履歴（スレッド間の短期記憶）

- 「さっきの◯◯」「昨日のあれ」など**過去スレッドの文脈が前提の依頼**は、まず
  `docs/AI_TASK_HISTORY.md`（直近2作業日分のエントリ）を読む。より古い経緯は `docs/history-archive/` を grep。
- 書き方・アーカイブの規約は [`docs/task-history.md`](docs/task-history.md)（追記する直前に読む）。

## ブランチ・PR の規律

- **push 前に、その PR がマージ済み/クローズ済みでないか必ず確認する**。正本は PR の実マージ状態
  （`merged`/`state`）。`git log origin/main..HEAD` は squash merge を検出できない。
  マージ済み/クローズ済みなら push せず、最新の `main` から新ブランチを切り直す。
- 新ブランチを切る前に必ず `git fetch origin main`（コンフリクト予防）。
- URL に影響する変更・外向き操作は必ずレビュー/確認を求める。

## 作業完了時の必須手順（ユーザーへの完了報告より前に・促される前に自分から）

1. `docs/AI_TASK_HISTORY.md` に「**なぜ**」を1エントリ追記し、3日分以上になっていたら古い日付分を
   アーカイブへ移す（規約: [`docs/task-history.md`](docs/task-history.md)）。
2. 作業中に実証で誤り・不足と分かったドキュメントをその場で正す（配布 doc なら outbox 提案で。
   黙って回避策で済ませない）。

具体的な手順は [`docs/ci-logs.md`](docs/ci-logs.md)。
