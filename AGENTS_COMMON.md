**作業前に必ず [`docs/OPS_SYNC_DESIGN.md`](docs/OPS_SYNC_DESIGN.md) を読むこと。** 共通基盤（ai-ops）の全体像（下り＝配布／上り＝提案、各ファイルの唯一の書き手、トークン構成）が書いてある。

## 作業完了時の必須手順（順番厳守・ユーザー返答より前に）

1. **タスク履歴を更新する**（このリポジトリの履歴ファイル）。残すのは「**なぜ**その変更をしたか」
   — コードを読めば分かる「何を変えたか」ではなく、コードに無い制約・判断根拠。
2. **再利用可能なノウハウを該当ドキュメントへ反映する**。特に**既存ドキュメントの手順が実証で
   誤り・不足だと分かったら、その場で正す**（黙って回避策で済ませない）。
   **ユーザーに促される前に、自分から完了手順の一部として行う。**
3. **共通基盤に関わる変更なら、クロスリポジトリ共通履歴にも残す**。`shared/` 由来ファイル・共通ブロックの
   配線・consumer-template 由来 workflow など、**複数リポジトリに同型で存在する「共通基盤」**を触ったときは、
   固有履歴だけだと別 consumer で同じ修正をするとき経緯が見えない。固有履歴に加えて、クロスリポジトリ履歴置き場
   （下記「クロスリポジトリ情報引き継ぎ」の Notion DB）にも「なぜ・どう直したか」を残す。
   **書き方は [`docs/cross-repo-history.md`](docs/cross-repo-history.md) に従う**（読む側の手元から辿れる手がかりの付け方）。

> 履歴ファイルの場所・保持件数など、リポジトリ固有の設定は各 AGENTS.md の固有パートで定義する。

## 新情報・ノウハウの記録先の判断

- **常時ロード層（AGENTS.md）に書く**: どのタスクでも必要なもの（デプロイ先・環境変数・外部連携の概要・
  ローカルで再現できない制約）。全タスクのコストになるので最小限に。
- **オンデマンド層（`docs/` 等）に書く**: 特定タスクでのみ必要なもの（手順・設計判断・障害調査）。
  常時ロード層には「**発火トリガ＋不可逆ガード**」だけ残し、詳細手順は `docs/` の該当ファイルへ逃がして
  そこを指す（抽象的な義務だけでは自主発火しないので、トリガは具体的に書く）。
- 判断基準: 「次のタスクが何であれ必要か？」NO なら常時ロード層に置かない。
- 新規追記だけでなく「**既存ドキュメントの実証的訂正**」も同格の対象。

## ブランチ・PR の規律

- **既存ブランチへ積み増してよいのは、その PR がまだ open（未マージ・未クローズ）のときだけ。**
  判断軸は「同じ依頼か」ではなく **PR の状態**。open な PR の続きなら積んでよい。
- **push 前に、その PR がマージ済み/クローズ済みでないか必ず確認する**（無条件）。正本は PR の実マージ状態
  （`merged`/`state`）。`git log origin/main..HEAD` は squash merge を検出できないので正本にしない。
  **マージ済み/クローズ済みなら push せず、最新の `main` から新ブランチを切り直す。**
- **新ブランチを切るときは必ず先に `git fetch origin main`**（コンフリクト予防）。
- URL に影響する変更・外向き操作は必ずレビュー/確認を求める。

## クロスリポジトリ情報引き継ぎ（Notion）

- **1セッション = 1リポジトリ。他リポジトリには一切アクセスできない**（読み取り・編集・`WebFetch`・push の
  いずれも不可。Claude Code の固定的な制約で、回避策は無い）。だから「他リポジトリを見れない／直せない」と
  毎回弁明しない。代わりに次のように動く。
- **別リポジトリでの作業が必要になったら、その依頼を Notion「AI Cross-Repo Task Log」に書く**
  ——そのリポジトリにスコープされたセッションで拾えるように。ユーザーの指示を待たず、自分から記録する。
- **依頼を書いたら、対象リポジトリごとに「コピペ用の指示文」もチャットに出す**（ユーザーが各リポの
  セッションへ貼れるように、対象が複数なら1つずつ）。各指示文は **冒頭に短い概要を1行**置く——Claude Code は
  最初の入力からセッションタイトルを生成するので、「Notion を読んで作業して」だけだと適切なタイトルにならない。
  概要に続けて、該当 Notion エントリの URL と「読んで作業して」を書く。
- **タスク固有の引き継ぎ**（「リポジトリ X と同じことを Y で」）にのみ Notion を使う。
  タスクに必要と判断したときは自律的に参照してよい。記録（書き込み）はユーザー指示時、別リポジトリでの作業依頼、
  または共通基盤の対応履歴を残す場合に行う。
- **共通基盤の対応履歴**のうちリポジトリをまたいで見えるべきものは Notion のクロスリポジトリ履歴 DB
  （"AI Cross-Repo Task Log"）に残す（書き方は [`docs/cross-repo-history.md`](docs/cross-repo-history.md)）。
  `.ai-ops/outbox/` は共通ブロックを置換する経路なので**履歴の記録には使わない**（共通ブロックを壊す）。
- 恒常的な共通運用ルールは Notion ではなく **ai-ops（このブロック・`docs/` 配下の共通 doc）** が正。手動リレーしない。
- **共通 doc（`docs/` 配下）の横断的・構造的な修正は Notion でエスカレーションしてよい**: 1点訂正は outbox 提案、
  複数 doc にまたがる・再構成が要る変更は Notion「AI Cross-Repo Task Log」に依頼を出して ai-ops 側で一括編集する
  （outbox 主・Notion 従。判断と書式は [`docs/outbox-proposal.md`](docs/outbox-proposal.md)）。

## この共通ブロック・共通 doc を直したくなったとき（重要）

マーカー区間（共通ブロック）も `docs/` 配下の共通 doc も、各リポジトリで手編集すると**次の同期で上書きされる**。
共通ルールの追加・修正に気づいたら、**作業中リポジトリの `.ai-ops/outbox/` に提案ファイルを置く**（マーカー区間・
配布 doc は触らない）。これが consumer→ai-ops の唯一の上り口で、転記もドリフトも起きない（ai-ops が単一の正のまま）。

- **書式・テンプレート・種別（`common-block-edit` / `shared-file`）の書き方は [`docs/outbox-proposal.md`](docs/outbox-proposal.md)。**
- 各リポジトリの `AGENTS.md` のマーカー区間や配布された `docs/` を直接書き換えない（上書きで消えるだけでなくドリフトの原因）。
- 横断的・構造的な修正は Notion 経由のエスカレーションでもよい（上の節）。

## 共通か固有かの判断（「これは他リポジトリでも要る」と気づいたとき）

作業中に再利用可能な知見・ルール・インフラに気づいたら、まず **「全リポジトリで同一内容が要るか？」** を問い、出し先を振り分ける:

- **全リポジトリ共通の振る舞いルール（テキスト）** → ai-ops が正本。`.ai-ops/outbox/` に `common-block-edit` 提案を置く。
  このリポジトリの `AGENTS.md` を直接書かない。
- **このリポジトリ固有**（他では違う／片方だけ必要）→ このリポジトリのドキュメントや `AGENTS.md` の固有パートに書く。
- **全リポジトリでバイト一致すべき実ファイル・共通 doc**（composite action・共有スクリプト・共通の手順 doc 等）
  → ai-ops の `shared/` が正本。`.ai-ops/outbox/` に `種別: shared-file` / `対象パス: <shared/ からの相対パス>` 提案を置く
  （consumer の同じパスへ再配布される）。このリポジトリだけに勝手に置かない（ドリフトの元）。
- **共通だが一部が固有** → 共通部分だけ上り提案し、固有部分はこのリポジトリに薄く残す
  （例: 「新規 workflow に publish-ci-logs を組み込む」義務は共通、collector 名やドキュメントの場所は固有）。

## CI ログ運用（新規ワークフローを追加するとき）

**新規に `.github/workflows/` を追加するときは必ず CI ログ出力（`publish-ci-logs`）を組み込む**（義務自体は全リポジトリ共通）。
具体的な手順（permissions・ログ出力先・`if: always()` の2ステップ・collector 登録）は
**[`docs/ci-logs.md`](docs/ci-logs.md) に従う**。collector・設計ドキュメントのファイル名や有無はリポジトリ固有。
