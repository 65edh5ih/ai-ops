**この区間は全リポジトリ共通ルール**（正本: ai-ops リポジトリの `AGENTS_COMMON.md`。仕組みの全体像は [`docs/OPS_SYNC_DESIGN.md`](docs/OPS_SYNC_DESIGN.md)）。

## 応答言語

- **ユーザーへの実質的な回答**（説明・分析・完了報告の本文）は、そのユーザーの言語で書く。
  回答本文が長いほど書き始めの言語（特に英語）に引きずられやすいので、最後まで揃える。
- 対象外: ツール前後の短い状況報告（英語のままでよい）、コミットメッセージ・コード内コメント・
  技術的識別子（プロジェクトの慣習に従う）。

## ユーザーへの操作指示は直リンク URL で

GitHub・Cloudflare 等の外部サービス上の設定作業をユーザーに依頼するときは、可能な限り
**目的の画面への直リンク URL**を提示する（メニューをたどる説明だけで済ませない。
例: GitHub の Actions secrets は `https://github.com/<owner>/<repo>/settings/secrets/actions`）。
URL を推測で組み立てたときはその旨を添える。
**長く残すドキュメントに書く場合は、URL に加えてメニュー経路も併記する**
（サイト構造が変わって URL が無効になっても辿れるようにする。チャットでの単発指示は URL だけでよい）。

## 接続できなかった URL の報告

- ネットワークアクセスがブロックされて失敗したら（403/407 等）、**完了報告に「接続できなかった URL」
  として URL（最低限ドメイン）と目的を必ず明記する**（ユーザーが許可リストに追加するかの判断材料）。
- 代替手段で目的を果たせた場合も省略しない（次回も同じ壁に当たる）。黙って回避だけで済ませない。

## 同期されるファイルを手で編集しない

このマーカー区間・`docs/` の配布 doc・`.ai-ops/sync-manifest.txt` に列挙されたファイルは ai-ops が自動同期する
（手編集しても次回同期で上書きされ、ドリフトの原因になる）。追加・修正に気づいたら
**作業中リポジトリの `.ai-ops/outbox/` に提案ファイルを置く**。これが consumer→ai-ops の唯一の上り口。
書式は [`docs/outbox-proposal.md`](docs/outbox-proposal.md)（実際に提案を書く直前に必ず読む）。

## リポジトリ横断の作業（このリポジトリの外に手が要るとき）

- 原則1セッション=1リポジトリで、他リポジトリにはアクセスできない（毎回弁明しない）。例外は
  Claude Code on the web で**ユーザーが明示的に**リポジトリ追加を依頼した場合のみ（エージェント起点では
  追加しない。他のエージェントに同等機能は無い）。エージェント起点のリポジトリ横断は次のように動く。
- **他リポジトリの参照が必要になったとき**、セッションにリポジトリ追加ツール（`add_repo`）があれば、
  「別リポジトリなので読めません」と答える代わりに「`<owner>/<repo>` をこのセッションに追加して
  参照しますか？」と**はい/いいえで答えられる形で確認する**（選択式の質問ツールがあればそれを使う）。
  「はい」を得てから追加する（対象リポジトリ名を明示した確認への承諾＝ユーザーの明示依頼。
  無断では追加しない）。ツールが無い環境・ユーザー不在の自律実行では下の task 経路。
- **別リポジトリでの作業が必要になったら**、`.ai-ops/outbox/` に `種別: task` の依頼を置く。
  ai-ops 経由で対象リポジトリの `.ai-ops/tasks/` へ配布され、そのリポジトリのセッションが実行する。
  書き方・依頼後にユーザーへ伝えること: [`docs/cross-repo-tasks.md`](docs/cross-repo-tasks.md)。
- **`.ai-ops/tasks/` にファイルがあれば、それは承認済み依頼**（届いている＝オーナーがマージ済み）。
  現在のユーザー依頼と両立するなら対応し、完了したら `種別: task-done` 提案で消化を報告する（同 doc）。
  両立しないなら、存在をユーザーに伝えるに留める。

## 再利用可能な知見・ルール・ファイルに気づいたときの振り分け

「**全リポジトリで同一内容が要るか？**」（共通/固有）と「**どのタスクでも必要か？**」（常時/オンデマンド）の2軸で出し先を決める:

- 共通の**振る舞いルール** → `種別: common-block-edit` 提案（このマーカー区間を直接書かない）。
- 共通で**バイト一致すべき実ファイル・手順 doc**（composite action・共有スクリプト・`docs/` の共通 doc）
  → `種別: shared-file` 提案（このリポジトリだけに勝手に置かない）。
- **このリポジトリ固有** → このリポジトリの AGENTS.md 固有パートや docs に書く（共通だが一部固有なら共通部分だけ提案）。
- **常時 vs オンデマンドの高さも分ける**: 常時ロード層（AGENTS.md）は「発火トリガ＋不可逆ガード＋ポインタ」だけ、
  詳細手順・設計判断・障害調査は `docs/`（オンデマンド層）へ。「次のタスクでも必要か？」NO なら常時層に置かない。
- **オンデマンド層に doc を新規追加したら、そこへ辿り着く導線も同時に用意する**（常時層に発火トリガの
  1行を足す・関連するファイル操作ルールから参照させる 等）。導線の無いオンデマンド doc は誰も開かず、
  存在しないのと同じ（コストだけ増える）。共通 doc の導線も `common-block-edit` 提案で足す。

## 手順系 doc（SOP）の書式

エージェントに実行させる手順 doc を**新規作成・改訂するとき**（共通・リポジトリ固有を問わない）は、
直前に書式規約 [`docs/sop-format.md`](docs/sop-format.md) を読み、それに従う。既存の手順 doc は
触れたついでに準拠させる。

## ドキュメントは現在形で書く

- `docs/` の doc は、障害記録（incidents）・タスク履歴とそのアーカイブを除き、**現在の姿だけ**を書く。
  実態が変わったら該当箇所をその場で書き換える（過去の姿は git とタスク履歴に残る）。
  「旧パスは読み替えてください」のような読み替え表・移行バナーや、「YYYY-MM-DD: 追加運用」のような
  日付見出しでの追記をしない（同じ事実の新旧が併存し、どれが現行か読めなくなる）。
- 同じ事実（パス・手順・制約）の正本は1つに保ち、他の doc からはリンクする。
- doc を追加・削除・移動したら、参照元も同一 PR で更新する。

## タスク履歴（スレッド間の短期記憶）

- 「さっきの◯◯」「昨日のあれ」など**過去スレッドの文脈が前提の依頼**は、まず
  `docs/AI_TASK_HISTORY.md`（直近2作業日分のエントリ）を読む。より古い経緯は `docs/history-archive/` を grep。
- 書き方の規約は [`docs/task-history.md`](docs/task-history.md)（追記する直前に読む）。

## ブランチ・PR の規律

- **push 前に、その PR がマージ済み/クローズ済みでないか必ず確認する**。正本は PR の実マージ状態
  （`merged`/`state`）。`git log origin/main..HEAD` は squash merge を検出できない。
  マージ済み/クローズ済みなら push せず、最新の `main` から新ブランチを切り直す。
- 新ブランチを切る前に必ず `git fetch origin main`（コンフリクト予防）。
- URL に影響する変更・外向き操作は必ずレビュー/確認を求める。

## 作業完了時の必須手順（ユーザーへの完了報告より前に・促される前に自分から）

1. `docs/AI_TASK_HISTORY.md` に「**なぜ**」を1エントリ追記する（規約:
   [`docs/task-history.md`](docs/task-history.md)。アーカイブは ai-ops の自動バッチが行うので手を出さない）。
2. 作業中に実証で誤り・不足と分かったドキュメントをその場で正す（配布 doc なら outbox 提案で。
   黙って回避策で済ませない）。

## CI ログ運用

新規に `.github/workflows/` を追加するときは必ず CI ログ出力（`publish-ci-logs`）を組み込む。
具体的な手順は [`docs/ci-logs.md`](docs/ci-logs.md)。

## D1 データベースを扱うとき

新しい D1 データベースを作る、または既存 D1 に定常的な書き込み（ログ等）を実装するときは、
必ず [`docs/d1-capacity.md`](docs/d1-capacity.md) を読むこと。D1 の 5GB 容量上限は
**アカウント内「全 D1 の合計」**で複数リポジトリ横断の制約になるため、容量の測り方と
retention 実装のハマりどころ（バイトでなく行数で刈る・`OFFSET` 全スキャン禁止 等）を先に把握する。
